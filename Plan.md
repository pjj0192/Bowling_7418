# Plan

볼링 점수 계산기(`Game` 클래스)를 RED-GREEN-REVIEW 사이클로 구현한다.
각 항목은 하나의 사이클(RED 커밋 -> GREEN 구현 -> REVIEW 승인 후 커밋)에 대응한다.

## Cycle 1 (완료)

- **테스트**: 모든 투구가 거터볼(0핀)이면 총점은 0점이다.
- **왜**: `roll`/`score`의 가장 단순한 골격을 세우는 케이스. 아직 스페어/스트라이크 로직 없이도 통과 가능한 최소 시작점.
- **결과**: `roll()`이 핀 수를 기록하고 `score()`가 단순 합산하는 구현으로 통과. 리뷰 승인 후 커밋됨.

## Cycle 2 (완료)

- **테스트**: 한 번의 스페어(예: 5+5) 뒤의 다음 투구(예: 3)가 해당 프레임의 보너스로 더해진다.
- **왜**: 현재 구현(단순 합산)은 보너스 개념이 없어 이 테스트에서 실패해야 하며, 스페어 감지 및 보너스 가산 로직을 도입하게 만드는 케이스.
- **결과**: 프레임 순회 + 스페어 감지 로직으로 통과. 스트라이크는 아직 미구현(테스트 없음). 리뷰 승인 후 커밋됨.

## Cycle 3 (진행 중)

- **테스트**: 첫 투구로 스트라이크(10핀)면, 다음 두 번의 투구 합이 해당 프레임 보너스로 더해진다.
- **왜**: 현재 구현은 스트라이크를 스페어처럼 2개 투구를 소비하는 프레임으로 취급해 인덱스가 어긋나므로, 실제로 실패해야 하며 스트라이크 감지 및 1칸만 소비하는 로직을 도입하게 만드는 케이스.
- **투구 시퀀스**: 10(스트라이크), 3, 4, 이후 16번의 거터볼 (총 19투구)
- **기대 점수**: 프레임1 = 10+3+4(보너스) = 17, 프레임2 = 3+4 = 7, 나머지 0 → 총 24점
- **RED 결과**: 예상대로 어긋났으나 값 불일치가 아니라 `IndexError: list index out of range`로 실패함 — 스트라이크 프레임에서도 2칸씩 소비하다 보니 인덱스가 배열 끝을 넘어감. 스트라이크 로직 부재가 원인이므로 정상적인 RED로 판단.
